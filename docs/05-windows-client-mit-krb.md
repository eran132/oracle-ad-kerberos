# 05 · MIT Kerberos for Windows + `krb5.ini`

This chapter sets up the Kerberos client on the **Windows 11 host** so that `kinit`, `klist`, `kvno` work and so that DBeaver's JVM can find the realm configuration.

There are two viable Kerberos providers on Windows:

| Provider | Pros | Cons |
|---|---|---|
| **Native Windows SSPI** (built-in) | No install. Uses your interactive Windows login if the host is domain-joined. | Works cleanly only on a domain-joined host. JDBC needs the LSA tweak ([06](06-windows-lsa-and-ccache.md)) to read the TGT. Cache layout is opaque. |
| **MIT Kerberos for Windows (KfW)** *(this lab uses this)* | Identical CLI to Linux (`kinit`, `klist`, `kvno`). Works on a non-domain-joined Windows host (your VirtualBox host). Reads a plain-text `krb5.ini`. Trivial to debug. | Extra install. Need to set `KRB5CCNAME` to share the cache with DBeaver. |

Because the Windows host is **not** joined to `mylab.local` (the AD lives only inside VirtualBox), MIT KfW is the right choice for this lab.

---

## 1. Install MIT Kerberos for Windows

1. Download the 64-bit MSI from MIT: <https://web.mit.edu/kerberos/dist/index.html> → "Kerberos for Windows 4.x" → `kfw-4.1-amd64.msi` (or current).
2. Run the MSI. Default install location: `C:\Program Files\MIT\Kerberos\`.
3. Reboot if prompted — installer modifies `PATH` so `kinit.exe`, `klist.exe`, `kvno.exe` are on `PATH` system-wide.
4. Verify in a **new** PowerShell window:

   ```powershell
   PS> Get-Command kinit, klist, kvno | Select-Object Source
   PS> klist -V   # should print MIT Kerberos version, NOT the Windows built-in klist
   ```

   > Watch out: Windows ships its own `klist.exe` in `C:\Windows\System32`. If `Get-Command klist` resolves there instead of `C:\Program Files\MIT\Kerberos\bin\klist.exe`, MIT's `bin` dir isn't first on `PATH`. Either fix `PATH` order or always invoke MIT's tools by full path.

## 2. Place `krb5.ini`

MIT KfW reads, in order:

1. `%KRB5_CONFIG%` (if set)
2. `C:\ProgramData\MIT\Kerberos5\krb5.ini`
3. `C:\Windows\krb5.ini` (legacy, avoid)

Use option (2) — system-wide and survives Windows user changes:

```powershell
PS> New-Item -ItemType Directory -Path C:\ProgramData\MIT\Kerberos5 -Force | Out-Null
PS> Copy-Item .\config\windows\krb5.ini.example C:\ProgramData\MIT\Kerberos5\krb5.ini
```

The template in [../config/windows/krb5.ini.example](../config/windows/krb5.ini.example) is a 1:1 port of the server-side [../../tableau_ad_oracle/config/krb5.conf.example](../../tableau_ad_oracle/config/krb5.conf.example):

```ini
[libdefaults]
  default_realm = MYLAB.LOCAL
  dns_lookup_realm = false
  dns_lookup_kdc = true
  rdns = false
  forwardable = true
  default_tkt_enctypes = aes256-cts-hmac-sha1-96 aes128-cts-hmac-sha1-96
  default_tgs_enctypes = aes256-cts-hmac-sha1-96 aes128-cts-hmac-sha1-96
  permitted_enctypes   = aes256-cts-hmac-sha1-96 aes128-cts-hmac-sha1-96

[realms]
  MYLAB.LOCAL = {
    kdc = ad1.mylab.local
    admin_server = ad1.mylab.local
  }

[domain_realm]
  .mylab.local = MYLAB.LOCAL
  mylab.local  = MYLAB.LOCAL
```

Why these exact enctypes: the keytab generated by `ktpass -crypto AES256-SHA1` ([../../tableau_ad_oracle/scripts/ktpass-keytabs.ps1](../../tableau_ad_oracle/scripts/ktpass-keytabs.ps1)) only contains AES keys. Listing RC4 here would let the Windows client negotiate RC4 in a way the server cannot satisfy, and you would get `KRB_AP_ERR_MODIFIED` at ticket-use time — see [troubleshooting.md#enctype-mismatch](../troubleshooting.md#enctype-mismatch).

## 3. Hosts/DNS sanity

The Windows host needs to resolve `ad1.mylab.local` and `ora01.mylab.local`. If you're not pointing at the AD DNS server, add to `C:\Windows\System32\drivers\etc\hosts`:

```
192.168.56.10  ad1.mylab.local  ad1
192.168.56.20  ora01.mylab.local ora01
```

Notes:
- `dns_lookup_kdc = true` in `krb5.ini` means MIT KfW will try SRV `_kerberos._udp.MYLAB.LOCAL`. With AD DNS reachable that works automatically; with only hosts-file entries it falls back to the explicit `kdc = ad1.mylab.local` line. Either works.
- Setting `rdns = false` is important: reverse DNS for `192.168.56.20` typically gives nothing useful and a wrong PTR will break ST acquisition.

## 4. First `kinit`

```powershell
PS> kinit alice@MYLAB.LOCAL
Password for alice@MYLAB.LOCAL: ********
PS> klist
Ticket cache: API:Initial default ccache
Default principal: alice@MYLAB.LOCAL

Valid starting       Expires              Service principal
05/13/2026 09:14:02  05/13/2026 19:14:02  krbtgt/MYLAB.LOCAL@MYLAB.LOCAL
        renew until 05/20/2026 09:14:02
```

If `klist` shows the Windows native cache layout (`Cached Tickets: (...)` with LogonId), `klist` is resolving to `C:\Windows\System32\klist.exe`, not MIT's. Fix `PATH` and try again — see chapter [06](06-windows-lsa-and-ccache.md) for cache routing.

## 5. Test you can reach the Oracle service

```powershell
PS> kvno oracle/ora01.mylab.local@MYLAB.LOCAL
oracle/ora01.mylab.local@MYLAB.LOCAL: kvno = 2
```

A successful `kvno` here means:
- the SPN exists in AD on `svc-ora01` ([troubleshooting.md#kdc-err-s-principal-unknown](../troubleshooting.md#kdc-err-s-principal-unknown))
- the realm name and enctypes match
- you have a valid TGT in the cache that DBeaver can also see

You are now ready for [06 · LSA and credential cache](06-windows-lsa-and-ccache.md).
