# RECIPE — what is actually deployed to make this work

> A one-shot checklist of every concrete object, file, parameter, jar, env var, and registry key that has to exist for an AD user to open DBeaver on a Windows host, hit ORCLPDB1, and run queries as themselves. The numbered chapters in [docs/](docs/) explain *why* each piece is there; this file is the *what*.
>
> Validated 2026-05-14 against Oracle 19c (19.30), DBeaver Community 25.3.4, Windows Server 2022 AD, Win11 client.

---

## End state, in one diagram

```
  MYLAB\alice (Windows logon)
        ↓ Windows LSA holds AD TGT
  DBeaver (Java 21, ojdbc8 23.3 + 3 companion jars, JVM args set)
        ↓ JDBC thin, Kerberos AP-REQ for oracle/ora01.mylab.local@MYLAB.LOCAL
  ora01:1521  (sqlnet KERBEROS5 + /etc/oracle/keytabs/ora01.keytab)
        ↓ "ALICE@MYLAB.LOCAL" exists as IDENTIFIED EXTERNALLY user
        ↓ AFTER LOGON trigger fires ad_sync.refresh_user_roles
        ↓ DBMS_LDAP → ad1:636 with /u01/app/oracle/cmu/wallet → memberOf
        ↓ GRANT ORA_READERS_ROLE TO "ALICE@MYLAB.LOCAL"  (idempotent)
  Session live with role(s); SELECT works.
```

## 2 · Oracle host (`ora01.mylab.local` — RHEL 9 + Oracle 19c CDB `ORCLCDB`, PDB `ORCLPDB1`)

### 2a · OS-level prerequisites

| What | Path / Value |
|---|---|
| `/etc/hosts` | `192.168.56.10 ad1.mylab.local ad1` · `192.168.56.20 ora01.mylab.local ora01` |
| Time sync | `chronyd` peering with `ad1`; absolute skew < 300 s required (Kerberos 5-minute window) |
| Trusted CA at OS level | `/etc/pki/ca-trust/source/anchors/mylab-root-ca.pem` + `update-ca-trust extract` |
| Kerberos config | `/etc/krb5.conf` — default realm `MYLAB.LOCAL`, `kdc = ad1.mylab.local`, `default_ccache_name = KEYRING:persistent:%{uid}` |
| Service keytab | `/etc/oracle/keytabs/ora01.keytab` — 0640 `oracle:oinstall`, holds `oracle/ora01.mylab.local@MYLAB.LOCAL` (AES-256), generated by `ktpass` on the DC and `scp`'d over |
| SELinux | Default `enforcing` is fine for the production setup — the ad_sync path doesn't run into any AVC denial (proven during debugging) |
| firewalld | Inbound 1521/tcp open; outbound unrestricted (default) |

### 2b · Oracle Net configuration files (`$ORACLE_HOME/network/admin/`)

**`sqlnet.ora`** (the deployed state, exact):

```ini
SQLNET.AUTHENTICATION_SERVICES = (BEQ, KERBEROS5)
SQLNET.KERBEROS5_CONF = /etc/krb5.conf
SQLNET.KERBEROS5_CONF_MIT = TRUE
SQLNET.KERBEROS5_KEYTAB = /etc/oracle/keytabs/ora01.keytab
SQLNET.FALLBACK_AUTHENTICATION = FALSE
SQLNET.AUTHENTICATION_KERBEROS5_SERVICE = oracle

WALLET_LOCATION = (SOURCE = (METHOD = FILE) (METHOD_DATA = (DIRECTORY = /u01/app/oracle/cmu/wallet)))
```

> The wallet is here for the **ad_sync package's DBMS_LDAP outbound bind to AD**, not for CMU. CMU is disabled (see DB params below).

**`ldap.ora`** — not used for anything in the working setup. Optional; can be removed. (Was tried for CMU; CMU is off.)

**`tnsnames.ora`** — standard `ORCLPDB1` entry pointing to `ora01.mylab.local:1521`, unchanged from baseline lab.

**`listener.ora`** — unchanged; baseline Oracle listener.

### 2c · Oracle wallet (`/u01/app/oracle/cmu/wallet/`)

| File | Owner / Mode | Purpose |
|---|---|---|
| `cwallet.sso` | `oracle:oinstall` 0600 | Auto-login; server processes read this without prompting for a password |
| `ewallet.p12` | `oracle:oinstall` 0600 | Encrypted wallet — used by `orapki` / `mkstore` when the password is provided |
| `cwallet.sso.lck`, `ewallet.p12.lck` | 0600 | Wallet locks (created automatically) |

Wallet **content**:
- Trusted certificate: `CN=mylab-root-ca,DC=mylab,DC=local` (added via `orapki wallet add -trusted_cert`)
- Secret-store entries (added via `mkstore -createEntry`):
  - `ORACLE.SECURITY.USERNAME` = `svc-ora-ldap`
  - `ORACLE.SECURITY.DN` = `CN=svc-ora-ldap,CN=Users,DC=mylab,DC=local`
  - `ORACLE.SECURITY.PASSWORD` = the AD password for `svc-ora-ldap`

> The wallet construction recipe is at the top of [`scripts/oracle/ad-sync-install.sql`](scripts/oracle/ad-sync-install.sql) as a comment, and as an executable build script in the sibling repo.

### 2d · Database parameters (CDB + PDB, in `orclpdb1`)

| Parameter | Value | Why |
|---|---|---|
| `LDAP_DIRECTORY_ACCESS` | `NONE` | CMU disabled — see [docs/16](docs/16-cmu-19c-failure-mode.md) |
| `LDAP_DIRECTORY_SYSAUTH` | `NO` (`SCOPE=SPFILE`, applied after bounce) | CMU disabled |
| `OS_AUTHENT_PREFIX` | `''` (empty string) | Allows `IDENTIFIED EXTERNALLY` users named like `<UPN>` without an `OPS$` prefix |
| `REMOTE_OS_AUTHENT` | `FALSE` | Default; do not change |
| `SEC_CASE_SENSITIVE_LOGON` | `TRUE` | Default; needed so `ALICE@MYLAB.LOCAL` and the principal match exactly |

### 2e · Database objects (created by [`scripts/oracle/ad-sync-install.sql`](scripts/oracle/ad-sync-install.sql))

| Object | Owner | Purpose |
|---|---|---|
| Role `ORA_READERS_ROLE` | — | `CREATE SESSION` + `SELECT ANY TABLE`. Mapped to AD group `oracle-readers`. |
| Role `ORA_WRITERS_ROLE` | — | `CREATE SESSION` + `SELECT/INSERT/UPDATE/DELETE ANY TABLE`. Mapped to AD group `oracle-writers`. |
| User `AD_SYNC` (`NO AUTHENTICATION`) | itself | Schema that owns the sync code. Login-less. Granted: `CREATE PROCEDURE`, `UNLIMITED TABLESPACE`, `EXECUTE ON DBMS_LDAP`, `SELECT ON DBA_USERS`, `SELECT ON DBA_ROLE_PRIVS`, `CREATE USER`, `ALTER USER`, `DROP USER`, `GRANT ANY ROLE`, `GRANT ANY PRIVILEGE`. |
| Network ACL | (SYS) | `DBMS_NETWORK_ACL_ADMIN.append_host_ace` for host `ad1.mylab.local`, privileges `connect,resolve`, principal `AD_SYNC`. Without this you get `ORA-24247`. |
| Table `AD_SYNC.AD_SYNC_LOG` | `AD_SYNC` | Autonomous-tx log target. Query for sync trace. |
| Package `AD_SYNC.AD_SYNC` (spec + body) | `AD_SYNC` | Two entry points: `run` (full sync) and `refresh_user_roles(p_principal)`. Uses `DBMS_LDAP` over LDAPS-636 via the wallet. |
| Scheduler job `AD_SYNC.AD_SYNC_JOB` | `AD_SYNC` | `FREQ=MINUTELY; INTERVAL=10` — drift correction. |
| Trigger `SYS.AD_SYNC_ON_LOGON` | `SYS` | `AFTER LOGON ON DATABASE` — fires for every `*@MYLAB.LOCAL` connect; calls `refresh_user_roles`. |
| `EXECUTE ON ad_sync.ad_sync` granted to `PUBLIC` | — | So the SYS-owned trigger can call the package. |
| External users `ALICE@MYLAB.LOCAL`, `BOB@MYLAB.LOCAL`, `CAROL@MYLAB.LOCAL` | — | Auto-created by `ad_sync.run` on first invocation. Each is `IDENTIFIED EXTERNALLY` (no Oracle password). |

## 3 · Windows client (`wks01.mylab.local` — Win11, domain-joined to `MYLAB.LOCAL`)

> wks01 is a Vagrant-managed Win11 VM joined to the lab AD. The same recipe (minus the domain-join part) applies to a non-domain laptop, with the addition of MIT Kerberos for Windows + `KRB5CCNAME` env var — see [docs/05](docs/05-windows-client-mit-krb.md) and [docs/06](docs/06-windows-lsa-and-ccache.md).

### 3a · Host config

| What | Value / Setting |
|---|---|
| Domain membership | Joined to `MYLAB.LOCAL` (provisioned by `lab/provision/01-os-prep-and-join.ps1`) |
| Login | User logs in as `MYLAB\alice` (or bob/carol). Windows LSA fetches a TGT automatically. |
| Trusted Root CA | `mylab-root-ca` imported into `Cert:\LocalMachine\Root` so Windows trusts `ad1.mylab.local`'s LDAPS cert (used by some tooling; not strictly needed by DBeaver). |
| Registry — LSA TGT visibility | `HKLM\SYSTEM\CurrentControlSet\Control\Lsa\Kerberos\Parameters\allowtgtsessionkey = 1` (REG_DWORD). **Required** so JGSS-via-SSPI can read the LSA TGT from a user-mode JVM. Without this, DBeaver can't see the Kerberos session key. Reboot needed after setting. |
| Time | Windows w32time syncing to `ad1.mylab.local`; skew < 300 s |
| Hosts entries (optional fallback) | `192.168.56.10 ad1.mylab.local`, `192.168.56.20 ora01.mylab.local` |

### 3b · DBeaver installation

| What | Value |
|---|---|
| Version | DBeaver Community **25.3.4** (winget: `winget install dbeaver.dbeaver`) |
| Bundled Java | Java 21 (DBeaver 25.x bundles this — no separate JDK install) |

### 3c · Oracle JDBC jars (4 of them — drop into `C:\Program Files\dbeaver\drivers\oracle\`)

| Jar | Version | Purpose |
|---|---|---|
| `ojdbc8.jar` | **23.3.0.23.09** | Thin JDBC driver |
| `oraclepki.jar` | **21.9.0.0** | Kerberos5 handler + wallet support |
| `osdt_core.jar` | **21.9.0.0** | ASN.1 / crypto primitives referenced by `oraclepki` |
| `osdt_cert.jar` | **21.9.0.0** | X.509 certificate classes referenced by `oraclepki` |

> All four required. Drop one and you get `NoClassDefFoundError` at connect. The version split (`ojdbc8` 23.x vs `osdt_*` 21.9) is Oracle's own — they ship from different teams. See [docs/07](docs/07-dbeaver-oracle-driver.md).

### 3d · `dbeaver.ini` JVM args (in `C:\Program Files\dbeaver\dbeaver.ini` after the `-vmargs` line)

```ini
-Djava.security.krb5.conf=C:\ProgramData\MIT\Kerberos5\krb5.ini
-Djavax.security.auth.useSubjectCredsOnly=false
-Dsun.security.krb5.debug=false
-Dsun.security.jgss.debug=false
```

Plus the `--add-opens` set required by Java 17+ for Oracle JDBC's internal reflection:

```ini
--add-opens=java.base/java.io=ALL-UNNAMED
--add-opens=java.base/java.lang=ALL-UNNAMED
--add-opens=java.base/java.lang.reflect=ALL-UNNAMED
--add-opens=java.base/java.net=ALL-UNNAMED
--add-opens=java.base/java.nio=ALL-UNNAMED
--add-opens=java.base/java.security=ALL-UNNAMED
--add-opens=java.base/java.util=ALL-UNNAMED
--add-opens=java.base/sun.security.ssl=ALL-UNNAMED
--add-opens=java.base/sun.security.util=ALL-UNNAMED
--add-opens=java.security.jgss/sun.security.jgss=ALL-UNNAMED
--add-opens=java.security.jgss/sun.security.krb5=ALL-UNNAMED
--add-opens=java.security.jgss/sun.security.krb5.internal=ALL-UNNAMED
--add-opens=java.security.jgss/sun.security.krb5.internal.crypto=ALL-UNNAMED
--add-opens=java.security.jgss/sun.security.krb5.internal.ccache=ALL-UNNAMED
```

The five `java.security.jgss/*` lines are the non-obvious ones. Without them you get cryptic `IllegalAccessException` / `InaccessibleObjectException` at connect.

### 3e · `C:\ProgramData\MIT\Kerberos5\krb5.ini`

Even on a domain-joined host, Oracle JDBC reads this file via the `-Djava.security.krb5.conf` JVM arg above (Windows native Kerberos has its own config, separate). Critical content:

```ini
[libdefaults]
default_realm = MYLAB.LOCAL
forwardable   = false              # <-- the fix for "EncryptionKey: Key bytes cannot be null"
default_tkt_enctypes = aes256-cts-hmac-sha1-96
default_tgs_enctypes = aes256-cts-hmac-sha1-96

[realms]
MYLAB.LOCAL = {
  kdc            = ad1.mylab.local
  default_domain = mylab.local
}

[domain_realm]
.mylab.local = MYLAB.LOCAL
 mylab.local = MYLAB.LOCAL
```

> **`forwardable = false`** is the magic line. Default `true` triggers Oracle JDBC's `getKRBCredForDelegation` code path which has a bug ("EncryptionKey: Key bytes cannot be null"). Setting it to `false` skips delegation and the auth works. See [troubleshooting.md#jdbc-encryptionkey-null](troubleshooting.md#jdbc-encryptionkey-null).

### 3f · DBeaver connection (`data-sources.json`) — the four properties that matter

Connection properties block (DBeaver UI: *Connection Settings → Driver properties*):

| Property | Value | Why |
|---|---|---|
| `oracle.net.authentication_services` | `KERBEROS5` | **Bare**, no parens. JDBC driver does NOT recognise `(KERBEROS5)` despite SQL*Net using parens. |
| `oracle.net.kerberos5_mutual_authentication` | `false` | `true` triggers the `getKRBCredForDelegation` bug even with `forwardable=false` in some flows. |
| `oracle.net.kerberos5_cc_name` | `C:/Users/alice/krb5cc` | File ccache. Optional on domain-joined (LSA works), but explicit is more reliable. |
| `oracle.net.kerberos5_conf` | `C:/ProgramData/MIT/Kerberos5/krb5.ini` | Tells the driver where to read the Kerberos config from. Forward slashes for JDBC. |

`auth-model` field in `data-sources.json` must be `oracle_native` — **not** `oracle_os`. `oracle_os` causes DBeaver to send an empty password (ORA-01017 fallback).

Username field: **blank**. Password field: **blank**. Identity comes from Kerberos.

## 5 · Day-2 operations (one-liners)

| You want to… | Do this |
|---|---|
| Add a new AD user | `New-ADUser`, `Add-ADGroupMember oracle-readers <user>`. Wait ≤10 min, OR they log in once (trigger does it instantly). |
| Change a user's privileges | `Remove-ADGroupMember oracle-readers <user>; Add-ADGroupMember oracle-writers <user>`. Next sync flips them. |
| Add a new permission set | `CREATE ROLE …`, grants on it, edit `ad_sync.ad_sync.group_map()` to add the new pair, re-run [`scripts/oracle/ad-sync-install.sql`](scripts/oracle/ad-sync-install.sql). |
| See what the sync did | `SELECT TO_CHAR(ts,'HH24:MI:SS.FF3'), lvl, msg FROM ad_sync.ad_sync_log ORDER BY ts;` |
| Force a sync now | `BEGIN ad_sync.ad_sync.run; END;` |
| Disable the sync temporarily | `BEGIN DBMS_SCHEDULER.disable('AD_SYNC.AD_SYNC_JOB'); END;` + `ALTER TRIGGER sys.ad_sync_on_logon DISABLE;` |
| Rotate the `svc-ora01` keytab | See [docs/10 — Operations & rotation](docs/10-operations-rotation.md). |
| Rotate the `svc-ora-ldap` password | Reset in AD; update `ORACLE.SECURITY.PASSWORD` in the wallet via `mkstore -modifyEntry`; restart listener. |

## 6 · What was attempted but does NOT make it work

The full investigation is in [docs/16-cmu-19c-failure-mode.md](docs/16-cmu-19c-failure-mode.md). Short version, so nobody re-tries them:

- ❌ Oracle CMU with shared-schema (group-DN mapping) and Kerberos — bug, `ORA-28030`, every documented config.
- ❌ Oracle CMU with exclusive-schema (per-user DN or UPN mapping) and Kerberos — same `ORA-28030`.
- ❌ `dsi.ora` vs `ldap.ora` swap — same failure (doc-recommended, doesn't help).
- ❌ SELinux permissive — not the cause.
- ❌ firewalld stopped — not the cause.
- ❌ Stripping corporate DNS forwarders from `/etc/resolv.conf` — not the cause.

The wallet, network path, AD permissions, and PL/SQL DBMS_LDAP all work in isolation; only Oracle's internal NTZ TLS code path inside the CMU subsystem fails (closes the TCP socket 7 ms in, no TLS Client Hello).
